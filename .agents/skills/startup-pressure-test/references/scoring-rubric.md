# Scoring Rubric

Use scores to make judgment concrete, not to create fake precision. Score conservatively when evidence is thin.

## Default Scorecard

Use these six rows for compact output:

| Area | 1/5 | 3/5 | 5/5 |
|---|---|---|---|
| Pain intensity | Mild annoyance or abstract desire | Clear frustration with some cost | Expensive, frequent, risky, or career/business-critical pain |
| Buyer clarity | Vague user or broad demographic | Plausible segment but buyer/user split unclear | Specific buyer/user with budget, authority, and repeated need |
| Urgency | Can wait indefinitely | Would solve eventually if convenient | Needs a fix now or by a recurring deadline |
| Current workaround | No visible workaround | Manual or imperfect workaround exists | Customer already pays, hires, hacks, or spends serious time |
| Differentiation | "Better/cheaper/easier" only | Some wedge but easy to copy | Clear wedge tied to workflow, data, distribution, speed, or trust |
| Speed to validate | Needs months or heavy build | Can test with prototype plus outreach | Can test this week with conversations, concierge, or pre-sell |

## Expanded Scorecard

Use when the user asks for deep analysis:

| Area | What To Check |
|---|---|
| Founder advantage | Domain access, credibility, distribution, technical edge, or lived pain |
| Distribution access | Existing channels, communities, warm intros, search demand, partnerships |
| Monetization | Budget owner, pricing basis, willingness to pay, existing spend |
| Retention potential | Recurring workflow, habit, data accumulation, integration depth |
| Defensibility | Proprietary data, network effects, workflow lock-in, brand, regulatory advantage |
| Timing | New regulation, platform shift, model capability, cost drop, market behavior change |

## Scoring Rules

- Tie every score to evidence or an explicit assumption.
- Penalize vague customers and vague buying motions.
- Penalize ideas that require scaled distribution before learning.
- Reward evidence of current spending or painful workarounds more than compliments.
- Reward fast validation even when the idea is weak; speed reduces downside.
- Do not average scores into a fake final number unless the user asks.

## Verdict Guidance

`Strong`:

- Specific painful problem.
- Reachable early adopters.
- Existing workaround or spend.
- Clear wedge.
- Fast test available.

`Weak`:

- Real-ish problem but low urgency, unclear buyer, weak differentiation, or hard acquisition.
- Continue only after narrowing ICP or finding stronger evidence.

`Pivot required`:

- No acute pain, no reachable buyer, no switching reason, or validation would require building too much first.
- Recommend a sharper customer, narrower workflow, different wedge, or service-first version.
