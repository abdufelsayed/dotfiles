# Startup Pressure Test Playbooks

Use these playbooks when the user requests a specific mode, deep analysis, or a more operational output than the compact default.

## Quick Verdict

Goal: decide whether the idea is worth the next validation cycle.

Steps:

- Identify the target customer and promised behavior change.
- Name the core assumption.
- Find the 1-3 most dangerous failure modes.
- Give a verdict: `Strong`, `Weak`, or `Pivot required`.
- Provide the next smallest real-world test.

Output:

- Verdict
- Core Assumption
- Biggest Risk
- Next Test

## Pressure-Test

Goal: find the reasons the idea fails before the founder spends serious build time.

Steps:

- Clarify the customer, pain, current behavior, and purchase/user action.
- Identify the riskiest assumption.
- Rank fatal risks by severity.
- Evaluate whether the problem creates enough urgency to change behavior.
- Evaluate founder-market fit only from evidence provided.
- State the pivot path if the idea is weak.

Rules:

- Lead with the dangerous flaw, not generic encouragement.
- Keep every flaw specific to the idea.
- Make every core assumption testable before building.
- Do not pad lists.

Output:

- Verdict
- Core Assumption
- Fatal Risks
- Founder-Market Fit
- Pivot Path

## Problem Validation

Goal: determine whether the problem is real, painful, frequent, and attached to a reachable buyer or user.

Steps:

- Define the concrete situation where the pain occurs.
- Identify who feels it most acutely.
- Identify what they do today and what that workaround costs.
- Write discovery questions about past behavior.
- Define validation criteria.
- Give a painkiller/vitamin verdict.

Discovery questions should ask:

- What happened last time?
- What did the customer do instead?
- How much time, money, risk, or embarrassment did it cost?
- Who else was involved?
- What would make them switch now?

Avoid:

- Would you use this?
- Would you pay for this?
- Do you like this idea?

Output:

- Specific Pain
- Early Adopter
- Current Workaround
- 5 Discovery Questions
- Validation Criteria
- Painkiller/Vitamin Verdict

## Competition Map

Goal: identify what the customer would need to abandon, not just named direct competitors.

Steps:

- Map current behavior.
- Map direct tools solving the same job.
- Map indirect alternatives: spreadsheets, agencies, internal tools, manual labor, status quo, marketplaces, assistants, or existing platforms.
- Assess switching cost and satisfaction with current behavior.
- Identify the wedge required to switch.

Rules:

- Current behavior is always competition.
- "No competition" means the customer may not care enough.
- Differentiation must be concrete: faster by X, removes Y workflow, reaches Z user, integrates with A, produces B measurable result.

Output:

- Current Behavior
- Direct Competitors
- Indirect Competitors
- Switching Cost
- Real Enemy
- Required Wedge

## First 10 Customers

Goal: produce a manual, specific path to the first real users or buyers.

Steps:

- Name the narrowest early adopter segment.
- Identify where they already gather.
- Use founder-led outreach, warm networks, communities, directories, events, and direct observation.
- Ask for conversations before selling.
- Define success as behavior: payment, repeated use, workflow change, referral, or active onboarding.

Rules:

- No paid ads as the first move.
- No generic mass outreach.
- No automation before learning what converts.
- Keep messages personal and tied to a real observed pain.

Output:

- Narrow ICP
- Where To Find Them
- Manual Outreach Approach
- First Message
- Success Criteria
- 10-Customer Path

## MVP Plan

Goal: design the smallest build/manual workflow that tests the riskiest assumption with real users.

Steps:

- State the single assumption being tested.
- Define the minimum experience needed to test it.
- Mark which parts can be concierge, manual, mocked, or no-code.
- Cut all features that do not test the assumption.
- Define pass/fail behavior.
- End with real user exposure.

Rules:

- A demo is not validation unless it changes behavior.
- Internal testing is not launch.
- The MVP should be smaller than the founder wants.
- Bundle fewer assumptions when possible.

Output:

- Core Assumption
- Minimum Test
- Build
- Fake/Manual
- Cut
- Pass/Fail Criteria
- 2-Week Plan

## Validation Sprint

Goal: turn the idea into a 1-2 week sequence of tests with decision gates.

Steps:

- Select the riskiest assumption.
- Define the test population.
- Define the offer, conversation, landing page, demo, concierge workflow, or prototype.
- Define metrics and minimum evidence threshold.
- Define kill/pivot/continue gates.

Output:

- Sprint Goal
- Assumption Under Test
- Daily Plan
- Evidence Threshold
- Kill/Pivot/Continue Gate
- Follow-Up Decision

## Investor-Style Memo

Goal: assess whether the idea is fundable or more likely a bootstrapped/service opportunity.

Steps:

- Evaluate market pull, timing, wedge, distribution, retention potential, and defensibility.
- Separate "good business" from "venture-scale startup".
- Identify what proof would change investor perception.

Output:

- Thesis
- Why This Could Work
- Why This Probably Fails
- Venture-Scale Risk
- Proof Needed
- Recommendation
